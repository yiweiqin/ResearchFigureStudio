# Stylist Stage

## Summary

Use this stage after the paper brief and reference-image slot analysis, but
before writing any image2 prompt. The goal is to make all generated image blocks
look like one coherent figure instead of unrelated illustrations.

## Required Output

Write `reference_style_profile.json` and `style_sheet.md` or
`style_sheet.json` in the output directory. The JSON profile is the machine
contract; the Markdown sheet is the human-readable summary.

Minimum fields:

- `style_name`: short name for the visual system
- `palette`: 4-8 colors with roles such as panel fill, accent, warning, cache,
  data, model, and neutral line
- `color_tokens`: exact reference-derived color records with `token_id`, `hex`,
  `rgb`, `hsl`, `source_region`, and `usage`; panel headers, fills, strokes,
  arrows, and local slot palettes must refer to these token IDs
- `reference_priority`: use `reference_image_primary`; paper concepts only
  adapt terminology and scientific meaning
- `line_style`: outline color, line weight, corner radius, connector style
- `shadow_style`: shadow softness, direction, opacity, and when to avoid shadow
- `viewpoint`: flat, slight isometric, front-facing card, or mixed
- `icon_complexity`: simple, medium, or detailed, with size rules
- `background`: transparent, white, paper tint, or flat color
- `visual_density`: expected density for macro panels, meso cards, and micro icons
- `content_fill_target`: default 88%-95% useful visual content per slot
- `margin_policy`: maximum pure empty margin, usually 10%-12% on any side
- `background_fill_strategy`: how background detail, card surfaces, or texture
  extend near the canvas edge without changing scientific meaning
- `icon_scale_rule`: micro icons should be large and full-frame, not tiny objects
  centered on a blank canvas
- `card_density_rule`: meso cards should use full-bleed card surfaces and
  supporting detail rather than sparse white panels
- `font_layer_rules`: font family or style for editable labels, minimum size,
  weight, and label placement rules
- `image2_text_policy`: none, decorative_only, or editable_overlay_required
- `forbidden_style_drift`: styles, colors, or rendering effects to reject

## Prompt Inheritance

Every image2 prompt must reference both:

- the relevant `slot_inventory` or `figure_program.json` slot entry
- the `reference_style_profile.json` fields and color token IDs that control
  palette, viewpoint, line style, density, background, and text policy
- the local `reference_slot_crops/<slot_id>.png` crop path used by the prompt
  planner

Do not let each prompt invent a new style. If a block needs a special treatment,
record the exception in the style sheet or in that slot's `style_override` field
inside `figure_program.json`.

## Text Rules

The style sheet controls what text image2 may create:

- `none`: no visible text
- `decorative_only`: tiny non-critical marks, unreadable UI strokes, or vague
  card headings are acceptable
- `editable_overlay_required`: all meaningful text is added by the PPTX layer by
  default; Lark or SVG text layers are only for explicit collaboration or vector
  export needs

Critical scientific text is never generated by image2. This includes method
names, module labels, formulas, variables, metrics, axis labels, branch labels,
panel IDs, citations, and numeric results.

## Acceptance

Before prompting, check that the style sheet is specific enough to prevent:

- inconsistent icon shape language
- mixed 2D and 3D perspectives without intent
- unrelated color palettes across slots
- unplanned text inside generated blocks
- visual density that makes some slots look unfinished or overcrowded
- undersized icons or cards with large blank canvas
- prompt language that asks for generous margins or small centered objects
